# S3 経由ドキュメントアップロード API/UI 対応

状態: done

## 背景

Allganize RAG Evaluation JA の benchmark runner で PDF corpus を `/documents` に `contentBase64` JSON として送信したところ、`000857916.pdf` が `HTTP content length exceeded 10485760 bytes` で失敗した。UI からの通常アップロードも同じ JSON body 経路を使うため、大きめのファイルで同様の制約に当たる可能性がある。

## 目的

S3 を介したアップロードを API として用意し、UI と benchmark runner が大容量ファイルを JSON body に載せずにドキュメント登録できるようにする。

## 範囲

- API: S3 upload handoff 用 endpoint、schema、service/store 連携、権限境界、OpenAPI。
- UI: ドキュメントアップロードを S3 handoff 経由へ変更し、失敗時の表示を維持する。
- Benchmark: corpus seed で S3 handoff API を使えるようにする。
- Infra: 必要な S3 policy、CodeBuild 環境、snapshot/assertion。
- Docs: 運用/API 説明の必要最小限の更新。

## 計画

1. 既存 API/UI/benchmark/infra のアップロード経路を調査する。
2. S3 handoff API の contract と認可方針を実装する。
3. UI の upload client と hook を S3 handoff 対応へ変更する。
4. benchmark corpus seed を S3 handoff 対応へ変更する。
5. infra/docs/tests を更新し、対象検証を実行する。
6. 作業レポート、commit、push、PR、PR コメントまで完了する。

## ドキュメント保守方針

API contract と運用フローが変わるため、`memorag-bedrock-mvp/docs/OPERATIONS.md` または API examples の関連箇所を確認し、必要に応じて更新する。不要な場合は作業レポートで理由を明記する。

## PR レビュー観点

- API route が auth middleware と permission check に含まれているか。
- 署名付き URL、S3 key、metadata による任意 object ingest や権限境界 bypass がないか。
- UI が大容量ファイルを base64 化して API に送らないか。
- benchmark 期待語句や dataset 固有分岐を RAG 実装に入れていないか。
- docs と実装、infra snapshot が同期しているか。

## リスク

- S3 署名付き URL と docs bucket の IAM が必要になり、CDK 差分が発生する。
- 既存の direct upload API 互換性を残す場合、旧経路の制限は残る。
- 大容量 PDF の実 end-to-end は AWS 環境依存のため、local test では署名付き URL contract と API/runner logic を中心に検証する。
