# FR-049 debug trace targetType の正本化

- 状態: done
- タスク種別: 修正
- 作成日: 2026-07-17
- 関連: Issue #358 / FR-049

## 背景

FR-049 は同期チャット処理を `ChatOrchestrationRun` として trace / store / contract 上で参照可能にする。一方、current main の chat orchestration producer は完成した回答 trace を `targetType: "rag_run"` として永続化し、型・schema は `rag_run` と `chat_orchestration_run` の両方を許容しているため、チャット処理の正本語彙が一意でない。

## なぜなぜ分析（修正前）

### 問題文

2026-07-17 時点の current main で、FR-049 が要求する同期チャット処理の persisted debug trace が `ChatOrchestrationRun` ではなく `rag_run` として分類され、要求・runtime producer・fixture の語彙が一致していない。

### 確定事実

- `REQ_FUNCTIONAL_049.md` は同期チャット処理を `ChatOrchestrationRun` として扱う。
- `chat-rag-orchestrator.ts` の完成 trace producer は `targetType: "rag_run"` を固定している。
- standalone `/search` の trace producer も `rag_run` を使用し、検索子 trace とチャット親 trace を同じ値で分類している。
- `DebugTraceTargetType` / `DebugTraceTargetTypeSchema` は両語彙を許容する。
- 一部 permission-revocation test は `chat_orchestration_run` を検索するが、正常な chat trace の targetType を直接固定していない。
- open PR #387 は `chat-rag-orchestrator.ts`、`memorag-service.ts`、schema、coverage、generated API docs と重複するため、取り込み時に再生成・再検証が必要である。

### 推定・未確定

- `rag_run` は当初 RAG 実装を表す汎用分類として導入され、後から追加された FR-049 の domain 名へ producer が追随しなかったと推定する。
- 既存 persisted item の `rag_run` が chat 親 trace か standalone search trace かを、全 item で安全に識別できる migration marker は未確認である。

### 根本原因

generic debug trace enum に複数の分類語彙を追加した一方、chat orchestration producer 専用の canonical target contract と回帰テストを設けなかったため、実装由来の `rag_run` と要求由来の `chat_orchestration_run` が並存した。

### 対策と影響範囲

- 新規 chat orchestration 親 trace は `chat_orchestration_run` に固定する。
- standalone search 子 trace は実体どおり `rag_run` を維持する。
- legacy の targetType 欠落は既存 v1 default の `rag_run`、明示済み値はそのまま読む。根拠なく runId / step から再分類する互換 shim は追加しない。
- 型、schema contract test、runtime producer、persistence/read fixture、FR-049 coverage、正本文書、decision record、generated docs を同期する。
