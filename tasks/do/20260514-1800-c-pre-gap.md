# C-pre-gap 調査タスク

状態: in_progress

## 背景

Wave 3-pre の C-pre-gap として、仕様 3B 章（ナレッジ品質・RAG利用可否）と既存 document store / RAG フィルタの差分、踏襲すべき既存挙動を調査し、docs メンテの PR を作成する。

## 目的

仕様 3B 章の実装ギャップ、既存挙動として維持すべき点、未確定事項を `docs/spec/gap-phase-c.md` に整理し、後続実装で benchmark 期待値や RAG 根拠性を壊さない判断材料にする。

## スコープ

- 対象: `docs/spec/gap-phase-c.md`, `reports/working/*c-pre-gap*.md`, `tasks/do|done/*c-pre-gap*.md`
- 調査対象: 仕様 3B 章、既存 document store、RAG フィルタ、chatrag-refusal-bench 期待値、minScore、answer support、citation、metadata budget
- 対象外: 実装変更、`docs/spec/CHAPTER_TO_REQ_MAP.md` の編集、PR merge

## タスク種別

調査

## 計画

1. 仕様 3B 章と関連 docs/spec-recovery を確認する。
2. document store / RAG フィルタ / benchmark / answer support / citation / metadata budget の既存実装・テストを読む。
3. confirmed / inferred / open_question / preserve を分けて `docs/spec/gap-phase-c.md` に整理する。
4. 指定検証を実行し、作業完了レポートを作成する。
5. commit / push / main 向け PR 作成、受け入れ条件コメント、セルフレビューコメントを行う。
6. PR コメント後に task md を done へ移動し、追加 commit / push する。

## ドキュメントメンテ方針

新規の恒久ドキュメントとして `docs/spec/gap-phase-c.md` を作成する。仕様章対応表は対象章が既に存在する前提のため編集しない。実装変更を伴わないため README や API 例は原則更新不要とし、必要性が見つかった場合のみ記録する。

## PR レビュー観点

- 仕様 3B 章の実装差分が confirmed / inferred / open_question / preserve に分離されているか。
- 後続実装で壊してはいけない RAG 品質・根拠性・拒否挙動が明記されているか。
- 実装変更や所有範囲外の編集を含んでいないか。
- 未実施の検証を実施済みとして書いていないか。

## リスク

- 並行作業中の他 worktree に未反映の最新実装・docs が存在する可能性があるため、本 PR は `origin/main` 時点の確認結果として扱う。
- 仕様 3B 章の文面に曖昧さがある場合は、推定で断定せず open_question として残す。
