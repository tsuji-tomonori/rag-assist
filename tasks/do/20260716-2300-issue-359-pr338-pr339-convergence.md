## 背景と問題文

PR #339 は古い main `e8ae57f6` を base に UsageEvent / pricing / completeness / export を115 filesで実装したまま open だが、同じ目的を current architecture に収束した PR #357 が main へ merge 済みである。PR #338 は同じ古い base から session-local evidence を42 filesで実装した partial PR で、現行 main の認可、document lifecycle、conversation history、generated docs へ機械適用できない。

本 Phase A は production code を変更せず、#339 の superseded 判断と #338 replacement の requirement/architecture trace を確定する。server 実装と Web 実装は小 PR 原則に従い Phase B / C の独立 task に分離する。

## 軽量なぜなぜ分析

- 問題文: 長期間 open の large PR が current main と意味的に重複または乖離し、単純 merge では現行契約と権限境界を壊しうる。
- 確認済み事実:
  - #339 は `UsageEvent`, pricing version, completeness, admin export を目的とする115-file patch である。
  - merged #357 は tenant-scoped usage event、versioned pricing、unknown/missing/unpriced completeness、usage/cost export permission・audit・expiry を current main に導入した。
  - #338 は `SessionDocumentContext`, `temporaryScopeIds`, previous citation anchor、Web attachment chip を導入した42-file patchだが、本文で MT-UI-003/004 と MT-TEMP-007/008 を未完と明記する。
  - current main は tenant+user partitioned conversation history、temporary attachment の tenant/owner/conversation/expiry retrieval boundary、通常 document list から chat scope を除く実装を既に持つ。
- 根本原因:
  - 古い PR の機能単位と current main の ownership・authorization・history・generated-docs contract が別々に進化し、PR 単位で収束可否を再判定する gate がなかった。
  - #338 の backend/Web/docs を一度に置換すると再び large PR になり、security と UI の review boundary が曖昧になる。
- 対策:
  - Phase A で evidence matrix と replacement decision を確定する。
  - Phase B で server-side session context/security、Phase C で history/UI を stacked PR として実装する。
