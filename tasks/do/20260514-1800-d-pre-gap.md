# D-pre-gap: 仕様 4B 章名称変更影響調査

状態: in_progress

## 背景

Wave 3-pre の D-pre-gap として、仕様 4B 章（チャット内オーケストレーション・ツール実行）の名称変更に先立ち、`Agent` を含む既存実装・仕様・ドキュメント上の影響範囲を調査する。

後続タスク `D-rename-agent-to-chat-orchestration` が安全に scope / scope-out を判断できるよう、実装変更は行わず、調査結果と踏襲すべき性能・品質チューニングを文書化する。

## 目的

- `Agent` を含む TypeScript 識別子、OpenAPI schema 名、ファイル名、docs 参照を棚卸しする。
- 名称変更対象と、意味上 `Agent` のまま残すべき対象を分ける。
- 既存性能・品質チューニングの踏襲リストを `preserve` として明記する。
- 後続 rename PR の scope / scope-out、open question を明確にする。

## タスク種別

調査

## 書き込み範囲

- 主対象: `docs/spec/gap-phase-d.md`
- 作業レポート: `reports/working/*d-pre-gap*.md`
- task 状態: `tasks/do/*d-pre-gap*.md`, `tasks/done/*d-pre-gap*.md`

`docs/spec/CHAPTER_TO_REQ_MAP.md` は対象章が既にある前提のため、原則編集しない。

## 計画

1. `rg` で `Agent` / `agent` / schema / ファイル名 / docs 参照を調査する。
2. 仕様 4B 章の既存文脈と関連する chat orchestration 実装・OpenAPI・docs を確認する。
3. `confirmed` / `inferred` / `open_question` / `preserve` に分けて `docs/spec/gap-phase-d.md` を作成する。
4. 最小検証として `python3 scripts/validate_spec_recovery.py docs/spec-recovery` と `git diff --check` を実行する。
5. 作業完了レポートを作成し、commit / push / PR 作成 / PR コメントを行う。
6. PR コメント後に task を `done` 化して追加 commit / push する。

## ドキュメント保守計画

実装変更はしない。仕様ギャップ調査の durable docs として `docs/spec/gap-phase-d.md` を新規作成する。README、API 例、運用手順、AGENTS.md は挙動・手順変更を伴わないため更新対象外とする予定。

## PR レビュー観点

- 後続 rename の scope / scope-out が過不足なく分離されているか。
- `Agent` を含む識別子・schema・ファイル名・docs 参照の棚卸しが、調査コマンドと照合可能か。
- 性能・品質チューニングの踏襲事項が実装変更なしで読み取れる形になっているか。
- 未確定事項が open question として残っているか。

## リスク

- `agent` は自然言語・ドメイン語・GitHub agent など別文脈にも現れるため、後続 rename の対象判定を誤る可能性がある。
- OpenAPI 生成物や過去レポートは参照量が多く、後続 task では生成元と成果物のどちらを rename 対象にするか追加判断が必要になる。
- 同時作業中の他エージェントの Phase D 文書更新が後から入る可能性があるため、この PR では指定書き込み範囲に限定する。
