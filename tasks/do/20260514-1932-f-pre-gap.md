## 背景

章別仕様 4A/4B では、チャット内 RAG・回答生成、チャット内オーケストレーション、`ChatToolDefinition` / `ChatToolInvocation`、マルチターン状態の分割が定義されている。一方、現行実装は `apps/api/src/chat-orchestration/` の graph / nodes と会話履歴 store を中心に段階的に整備されており、toolId registry と multi-turn 永続構造は未整理の可能性がある。

## 目的

仕様 4A/4B 章と現行 `chat-orchestration` nodes の toolId 対応・multi-turn 構造のギャップを調査し、Phase F 実装 task `F-chat-tool-registry-multiturn` の scope / scope-out と踏襲すべき既存挙動を docs に記録する。

## 調査範囲

- 仕様章: `docs/spec/2026-chapter-spec.md` 4A / 4B、および関連する 14A debug trace 記述。
- 実装参照: `apps/api/src/chat-orchestration/`, `apps/api/src/adapters/*conversation*`, `packages/contract/src/schemas/chat.ts`。
- docs: `docs/spec/gap-phase-f.md`, `docs/spec/CHAPTER_TO_REQ_MAP.md`, 関連 REQ / `docs/spec-recovery/`。

## スコープ

- 仕様と実装の差分を `confirmed` / `partially covered` / `missing` / `divergent` で分類する。
- 現行 nodes と後続 toolId 候補の対応表を作る。
- マルチターン状態の現行保持と仕様上の不足を整理する。
- plan にある ChatRAG follow-up 軽量化、required fact planning 汎化、policy computation 汎化、`verify-answer-support` 閾値、minScore filter、diversity 制約、context budget を踏襲対象として記録する。
- `docs/spec/CHAPTER_TO_REQ_MAP.md` と必要最小限の REQ / traceability を更新する。

## 含まない

- API / contract / runtime code の実装変更。
- `ChatToolDefinition` 型や registry の実装。
- conversation history store の schema 変更。
- benchmark suite の追加。
