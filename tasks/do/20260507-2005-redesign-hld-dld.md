# HLD / DLD 設計見直し

- 状態: doing
- 作成日: 2026-05-07
- 対象: `memorag-bedrock-mvp/docs/3_設計_DES/01_高レベル設計_HLD/DES_HLD_001.md`, `memorag-bedrock-mvp/docs/3_設計_DES/11_詳細設計_DLD/`

## 背景

`DES_HLD_001.md` のコンポーネント一覧が英語中心で分かりづらく、主要フローも質問応答、文書管理、管理、認証、コスト見積もりが混在している。

## 目的

要求カテゴリ、論理コンポーネント、詳細設計、処理フローの関係が追える設計ドキュメントへ整理する。

## スコープ

- HLD を要求カテゴリ別の設計索引として再構成する。
- 英語コンポーネントに日本語名と説明を追加する。
- 主要フローを目的別に分割する。
- HLD で列挙した主要コンポーネント群に対応する DLD を追加または整理する。
- ドキュメント更新に必要な検証と作業レポートを残す。

## 非スコープ

- アプリケーション実装コードの変更。
- 要求そのものの新規追加または変更。
- 既存 API contract の変更。

## 作業計画

1. 既存 HLD / DLD / 要求カテゴリを確認する。
2. HLD をカテゴリ別構造に書き換える。
3. 不足しているコンポーネント群の DLD を追加する。
4. Markdown 体裁、関連要求、既存 DLD との重複を確認する。
5. 検証、作業レポート、commit、PR、受け入れ条件コメント、セルフレビューコメントを行う。

## ドキュメント保守計画

- `memorag-bedrock-mvp/docs/DOCS_STRUCTURE.md` の REQ / ARC / DES 分離方針に従う。
- 要求は変更せず、設計側で要求カテゴリとの対応を明確化する。
- 既存 DLD と重複する内容は参照関係を明示し、詳細が分散しないようにする。

## 受け入れ条件

- [ ] HLD が要求カテゴリ別に論理コンポーネントと対応 DLD を示している。
- [ ] 英語コンポーネント名に日本語名、説明、責務境界が付いている。
- [ ] 「主要フロー」が何のフローか分かる名前で分割されている。
- [ ] HLD の主要コンポーネント群について、既存または新規 DLD への対応が追跡できる。
- [ ] 追加 DLD が入出力、処理手順、例外、テスト観点を持つ。
- [ ] ドキュメント変更に対する最小十分な検証を実行し、未実施項目を実施済み扱いしない。
- [ ] 作業完了レポートを `reports/working/` に保存する。
- [ ] PR 作成後に受け入れ条件確認コメントとセルフレビューコメントを日本語で記載する。

## 検証計画

- `git diff --check`
- Markdown 対象ファイルの見出し、関連要求、リンク相当のパス確認
- 利用可能なら `pre-commit run --files <changed-files>`

## PR レビュー観点

- HLD が詳細を書き込みすぎず、DLD への索引として機能しているか。
- 既存要求 ID と設計カテゴリが矛盾していないか。
- docs と実装の同期に不要な実装変更を含めていないか。
- RAG の根拠性、認可境界、benchmark 固有分岐に関する設計原則を弱めていないか。

## リスク

- 設計粒度を細かくしすぎると、今後の更新負荷が増える。
- 既存 DLD と新規 DLD の責務境界が曖昧だと、同じ内容が複数箇所に重複する。
