# PR #328 権限継承回帰修正

## 背景

PR #328 のレビューで、document group 権限継承の互換性と Web 経路の到達性に関する回帰リスクが指摘された。特に legacy `hasExplicitPolicy=false` child group の既存共有解釈、`listDocumentGroups()` の UI 用実効権限注釈、Web の親継承フォルダ作成、readOnly folder の resource-level 操作抑止を固定する必要がある。

## なぜなぜ分析サマリ

- confirmed: 旧作成ロジックでは共有設定の有無に関係なく `hasExplicitPolicy=false` が保存されていたため、既存 child group には child 自身の `visibility/sharedUserIds/sharedGroups/managerUserIds` を explicit policy として扱うべきデータが存在する。
- confirmed: PR #328 の継承 resolver は、親ありかつ `hasExplicitPolicy/policyId` が falsy の child を親継承として解釈し、legacy child の共有解釈を変えうる。
- confirmed: Web の新規フォルダ作成は常に `visibility` を送るため、API で用意した「visibility 省略時に親継承」という主経路に到達できない。
- confirmed: Web の操作可否は global feature permission に偏り、選択 folder の `effectivePermission=readOnly` を十分に見ていない。
- root cause: 新しい継承 semantics と既存保存データの互換境界、および API が返す resource-level permission と Web UI guard の接続が不足していた。
- remediation: legacy data を explicit として扱う互換推論、一覧レスポンスへの実効権限注釈、Web の inherit payload 省略、readOnly folder 操作抑止をテストで固定し、失敗箇所を実装修正する。

## 受け入れ条件

- [x] API: legacy `hasExplicitPolicy=false` + child shared は parent private に潰されず reader が読める。根拠: `memorag-service.test.ts` の追加テストと API test pass。
- [x] API: legacy `hasExplicitPolicy=false` + child private は parent shared を誤継承せず reader に漏えいしない。根拠: `memorag-service.test.ts` の追加テストと API test pass。
- [x] API: `listDocumentGroups(user)` が `effectivePermission`, `policySource`, `inheritedFromFolderId` を UI 用に返す。根拠: `memorag-service.test.ts` の追加テストと API typecheck pass。
- [x] Web: 親フォルダから継承する新規フォルダ作成時は `visibility/sharedGroups/managerUserIds` など explicit policy 項目を送らない。根拠: `DocumentWorkspace.test.tsx` の追加テスト pass。
- [x] Web: 選択 folder が `effectivePermission=readOnly` の場合、global write/delete/reindex permission が true でも upload/share/delete/reindex 操作が disabled になり handler が呼ばれない。根拠: `DocumentWorkspace.test.tsx` の追加テスト pass。
