# ドキュメント登録オンボーディングと文書一覧レイアウトを修正

保存先: `tasks/do/20260714-0033-document-upload-onboarding.md`

状態: do

タスク種別: 修正

## 背景

2026-07-13 に確認されたドキュメント管理画面では、フォルダも文書もない初期状態で絞り込み欄が残余高まで伸び、登録の主操作は保存先未選択により無効になっている。利用者は「フォルダ作成」「保存先選択」「ファイル選択」の順序と、フォルダ作成時の必須項目を画面から判断できない。フォルダ設定モーダルには登録、作成、編集、共有、履歴が同時に並び、名称以外の任意設定が初回操作に必要であるように見える。

前回の方針検討では `claude -p --model fable` との実質2往復目で、簡易作成を「ルート直下・名称のみ」に限定し、詳細設定を後から行う方針へ収束した。

## 目的

利用者が初期画面だけを見て、保存先の用意から単一ファイルのアップロードまで迷わず完了できるようにする。同時に、条件付き要素で文書一覧の Grid 配置がずれる不具合と、設定モーダルのカード伸長を再発しない構造へ直す。

## なぜなぜ分析

### 問題文

2026-07-13 のデスクトップ版ドキュメント管理画面で、フォルダ0件・文書0件・保存先未選択のとき、検索・絞り込み欄が文書一覧領域の大半まで縦に伸び、利用者が次の有効な操作と登録必須項目を判断できない。

### 確認済み事実

- `.workspace/20260713/` の2枚のスクリーンショットに、縦長の絞り込み欄、無効なアップロード操作、複数目的を混在させた設定モーダルが写っている。
- 最新 `origin/main` の `.document-file-panel` は5行 Grid だが、`uploadDisabledReason` があると直下要素が6個になる。
- 条件付き `<p>` により、`minmax(0, 1fr)` の行へ文書テーブルではなく絞り込みバーが自動配置される。
- 空状態の主要ボタンは `canUploadToDestination=false` のため無効で、別のアイコンのみの「フォルダを作成」を探す必要がある。
- `CreateDocumentGroupRequestSchema` で必須なのは `name` だけである。
- ルートへ `name` のみで作成したフォルダは作成者管理の `ownerDefault` として扱える。親あり・明示設定なしでは親の実効権限を継承する。
- 設定モーダルの2列 Grid は同一行の短いカードを長いカードの高さまで stretch する。
- アップロード操作に下向きの `download`、設定操作に `share` アイコンを使っている。

### 推定・未確認

- 推定: 無効な主操作、アイコンのみの代替、必須・任意の混在が複合し、初見利用者の手を止めている。
- 未確認: スクリーンショットを取得したデプロイ commit は特定していない。ただし最新 `origin/main` の JSX/CSS が観測された形状と一致する。
- 未確認: 実利用者を対象にしたタスク完了時間の計測は未実施。

### 因果ツリー

1. レイアウト破綻
   - 直下要素の順序へ Grid 行を暗黙依存させた。
   - 条件付き注意文を追加しても、named area や wrapper による構造固定がなかった。
   - 見た目の回帰を検知するテストがなかった。
2. 登録導線の行き止まり
   - システム前提の「保存先選択」を、利用者の仕事である「文書追加」より先に露出した。
   - 実行不能なアップロードを主操作として残し、有効な次操作をアイコンだけで提示した。
   - フォルダ作成とアップロードを別の画面領域へ分断した。
3. 入力負荷
   - API では名称以外が任意でも、作成フォームですべての管理項目を同時表示した。
   - 初回登録と高度なアクセス管理を同じモーダルに混在させた。
4. 意味の取り違え
   - 操作とアイコンの意味が一致せず、文字ラベルも常時表示されなかった。

### 根本原因

- 直接原因: 文書一覧 Grid が要素数とDOM順へ暗黙依存している。
- 局所要因: 初回登録が「保存先を先に管理する」内部モデル中心の情報設計である。
- 流出原因: 初期状態、条件付き注意文、320px、キーボード操作を一連で守る回帰テストが不足している。
- 組織要因: 文書登録の最短ユーザーフローと、任意のフォルダ管理項目を分離するUI契約が要件・受け入れ条件へ明記されていなかった。

### 全量対応方針

- Grid を named area または固定 wrapper へ変更し、絞り込みは内容高、文書一覧だけを残余高へ割り当てる。
- 文字付き主CTA `ドキュメントを追加` と専用ダイアログを導入する。
- 保存先は full 権限の既存フォルダ選択、またはルート直下の名称のみ作成で用意する。
- 新規作成成功後は保存先へ選択するが、非同期処理後のファイルピッカー自動起動は行わない。
- 単一ファイルの選択、保存先確認、アップロード、解析・検索利用可能状態を一連で表示する。
- フォルダ0件、選択フォルダ0件、絞り込み0件の回復導線を分ける。
- 設定モーダルは内容高で表示し、初回登録を設定モーダルへ依存させない。
- 主操作へ文字ラベルを付け、upload/settings の意味に合うアイコンへ直す。
- component test、typecheck、build、UI inventory、可能な範囲の手動表示確認で再発を検知する。

## 方針

- WCAG 2.2 AA と JIS X 8341-3:2016 AA を基準にする。
- 主CTAは44px級、その他の操作は最低24 x 24 CSS pxを維持する。
- 新規フォルダの簡易 payload は `{ name }` のみにし、親、説明、公開範囲、共有先、管理者を送らない。
- 作成済みフォルダの詳細は既存設定機能で後から変更できる。
- 本番表示値はAPI・props・component stateまたは明示的な空/権限/失敗状態に限定し、架空フォルダや共有候補を置かない。
- 単一ファイルのみを対象とし、複数ファイルの部分成功UIは今回追加しない。
- 既存API、権限境界、RAG取り込み処理は維持する。

- Grid修正が条件付き要素の有無とDOM順へ再び依存していないか。
