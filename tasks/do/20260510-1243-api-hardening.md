# API hardening task

状態: do

## 背景

`rag-assist` の API 実装について、Hono / TypeScript のベストプラクティス観点から、認証 fail-open、JSON error 契約、CORS、保護 route 管理、route handler の `any`、config validation、REST schema と contract の drift リスクが指摘された。

## 目的

MVP API の安全性、契約一貫性、型安全性を高め、運用時の設定漏れや認証漏れを起きにくくする。

## Scope

- `memorag-bedrock-mvp/apps/api` の Hono app、auth、config、route utility、route handler、関連テスト
- 必要に応じた `memorag-bedrock-mvp` docs / README / env sample
- 作業レポート、PR コメント、セルフレビュー

## Plan

1. 現行 app / config / auth / route utility / tests / docs を確認する。
2. production で auth / Cognito / CORS 等が fail-closed になる設定検証を追加する。
3. `HTTPException` / unknown route の JSON error 応答を統一する。
4. protected path 手動列挙を public allowlist 方式へ寄せる。
5. `looseRoute` と handler 側 `any` を可能な範囲で削減し、型逃げを局所化する。
6. REST schema と `packages/contract` の drift について、実装可能範囲の統一または明示的な追跡を行う。
7. API tests / typecheck / docs check などを実行する。
8. レポート、commit、push、PR、受け入れ条件コメント、セルフレビューを完了する。

## Documentation maintenance plan

- 環境変数や本番起動条件が変わる場合は README / docs / env sample を更新する。
- API error 契約や OpenAPI 出力への影響を確認する。
- ドキュメント更新が不要な項目は作業レポートで理由を記録する。

## 受け入れ条件

- [ ] production で `AUTH_ENABLED=false` または Cognito 必須設定不足のまま保護 API が admin fallback で起動しない。
- [ ] `HTTPException` と unknown route が `{ error: string }` JSON 応答に統一される。
- [ ] CORS origin が環境変数で制御され、production で `*` 固定にならない。
- [ ] 認証 middleware の適用範囲が public allowlist 方式または同等に安全な構成になる。
- [ ] route handler の `(c.req as any).valid(...)` を削減し、残る型逃げが局所化または理由付きになる。
- [ ] config の本番必須値 validation が追加され、不正値の silent fallback が少なくとも production で起きない。
- [ ] REST schema と oRPC contract の drift リスクについて、実装またはドキュメント上の追跡が追加される。
- [ ] 変更範囲に見合う API test / typecheck / docs check が実行され、結果が記録される。
- [ ] 作業レポート、PR 受け入れ条件コメント、セルフレビューコメントが日本語で残る。

## Validation plan

- `git diff --check`
- `npm --prefix memorag-bedrock-mvp run test -w @memorag-mvp/api`
- `npm --prefix memorag-bedrock-mvp run typecheck -w @memorag-mvp/api`
- OpenAPI/docs check は package scripts / Taskfile を確認して最小十分なものを選択する。

## PR review points

- production fail-closed の挙動が local/test の開発体験を壊していないか。
- public endpoint が `/health` / `/openapi.json` / 明示的な OPTIONS などに限定されているか。
- ErrorResponseSchema と実レスポンスが一致しているか。
- RAG の根拠性、認可境界、benchmark 固有値混入に悪影響がないか。

## Risks

- route typing の全面改善は変更量が大きく、全量を 1 PR で完了できない可能性がある。
- production config validation は既存 deploy 環境の未設定を顕在化させる可能性があるため、docs 更新が必要。

## 実施結果

- production fail-closed config validation、CORS allowlist、public allowlist auth、JSON error contract を実装した。
- route handler の `(c.req as any).valid(...)` を `validJson` / `validParam` に置換した。
- REST / oRPC drift は完全統合ではなく、README / API design / PR self-review の確認 gate として明記した。
- 検証は `reports/working/20260510-1300-api-hardening.md` に記録した。
