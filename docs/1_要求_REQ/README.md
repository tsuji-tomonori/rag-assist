## 目的

MemoRAG MVP が満たす条件を、実装状況と混同せず追跡する。個々の要求は 1 要件 1 ファイルとし、同じファイルに検証可能な受け入れ条件を記載する。

## 正規の配置

| 分類 | 保存先 | 内容 |
| --- | --- | --- |
| プロジェクト要求 | `01_プロジェクト要求_PROJECT/` | 目的、スコープ、優先順位 |
| 製品要求 | `11_製品要求_PRODUCT/` | 機能要求、非機能要求、技術・品質制約 |
| 横断受入基準 | `21_受入基準_ACCEPTANCE/` | 複数要件にまたがる受入・評価条件 |
| 変更管理 | `31_変更管理_CHANGE/` | baseline、置換、trace、未確定判断 |

要求の主分類は `11_製品要求_PRODUCT/01_機能要求_FUNCTIONAL/README.md`、2026-07 時点の権限・共有・RAG 要求集合と実装 gap は `11_製品要求_PRODUCT/REQUIREMENTS_BASELINE_202607.md` を正とする。

## 要求と実装状況の扱い

- 要求ファイルは「必要な振る舞い」を定義し、コードが存在するだけで実装済みとは判定しない。
- `confirmed`、`inferred`、`conflict`、`open_question` を使い、事実、推定、矛盾、未確定判断を区別する。
- 未実装または部分実装の要求は削除せず、baseline の gap 表から `tasks/todo/` の実装 task へ追跡可能にする。
- 実装完了の判定は、要求の受け入れ条件、対象コード、変更範囲に見合うテストの三者が揃った時点で行う。
- Superseded 要求は履歴として残し、置換先を明記する。

## 文書境界

- 達成すべき条件は `docs/1_要求_REQ/` に置く。
- 構造と重要判断は `docs/2_アーキテクチャ_ARC/`、実装可能な契約は `docs/3_設計_DES/` に置く。
- 現行の監視・検証手順は `docs/4_運用_OPS/21_監視_MONITORING/` に置く。
- OpenAPI、Web/infra inventory は生成コマンドを正とし、出力だけを `docs/generated/` に置く。

## 変更時の確認

1. 要求を原子的かつ検証可能に記述する。
2. 受け入れ条件、要求属性、前後 trace を同じ変更で更新する。
3. 実装・設計・運用との不一致を gap として記録し、未実装なら todo を作成または既存 todo に統合する。
4. `python3 scripts/validate_docs.py` と変更範囲に対応する検証を実行する。
