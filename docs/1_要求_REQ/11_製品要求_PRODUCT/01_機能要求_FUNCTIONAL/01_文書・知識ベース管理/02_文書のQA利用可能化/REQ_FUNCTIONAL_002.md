# 要件定義（1要件1ファイル）

- L0: `rag-assist / MemoRAG MVP 機能要件`
- L1主分類: `1. 文書・知識ベース管理`
- L2主機能群: `1.2 文書のQA利用可能化`
- L3要件: `FR-002`
- 関連カテゴリ: なし

## 要件

- FR-002: アップロードした文書は、利用者が取り込み状態を確認できる処理を経て、質問回答の検索対象として利用可能な状態になること。

## 受け入れ条件（この要件専用）

- AC-FR002-001: 文書アップロード開始後、UI は API 由来の状態に応じて準備、転送、取り込み job 作成、抽出、chunk 作成、embedding、index 反映、完了を識別可能に表示すること。
- AC-FR002-002: 取り込みが完了する前は、質問回答で利用可能になったと表示しないこと。
- AC-FR002-003: 取り込みに失敗した場合、失敗状態と API 由来の失敗種別または理由を表示し、成功扱いにしないこと。
- AC-FR002-004: 取り込みが完了した場合、文書一覧更新後に詳細、対象資料への質問、保存先フォルダのいずれかへ進めること。文書情報がまだ取得できない場合は、その状態を明示すること。
- AC-FR002-005: 質問回答で文書を利用するときは、`FR-057` と `FR-076` に基づく認証・資源認可および検索 scope を維持すること。

## 要件の源泉・背景

- 源泉: 旧 `docs/1_要求_REQ/README.md` に定義された原子要件を分割移管。
- 源泉: `FR-038` の非同期取り込み run と既存 `DocumentUploadState` の状態契約。
- 背景: 要求単位で差分レビュー・追跡・変更管理をしやすくするため、1要件1ファイル運用へ移行。
- 背景: ファイル転送の成功と、抽出・embedding・index 反映を終えた検索利用可能状態は同一ではないため、途中状態と失敗を利用者へ区別して示す必要がある。

## 要件の目的・意図

- 目的: 登録文書を検索・回答処理に利用可能な状態へ確実に変換する。
- 意図: アップロード成功と回答利用可能状態のギャップを、観測可能な進捗と最終状態で埋める。
- 区分: 機能要求として管理し、関連要件と独立に改訂可能な単位を維持する。
