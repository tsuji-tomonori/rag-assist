## 要件

- FR-021: 利用者は回答不能な質問を担当者向け問い合わせとして登録できること。

## 受け入れ条件（この要件専用）

- AC-FR021-001: 利用者は問い合わせのタイトル、質問本文、依頼者情報、担当部署、優先度を送信できること。
- AC-FR021-002: システムは登録済み問い合わせを一意の ticket として作成し、元の質問本文または回答不能理由を参照できる形で保存できること。
- AC-FR021-003: システムは問い合わせ作成者の認証済み userId を保持し、本人向け詳細取得や本人解決済み化の所有者境界に使えること。
- AC-FR021-004: 通常利用者が問い合わせを送信した後に、担当者一覧や debug trace 一覧の取得に依存せず送信完了を表示できること。
- AC-FR021-005: 通常利用者の問い合わせ登録は、問い合わせ一覧参照、回答登録、debug trace 詳細参照の権限を要求しないこと。

## 要件の源泉・背景

- 源泉: `reports/working/20260501-0034-assignee-answer-ui-ux.md`、`reports/bugs/20260502-1135-question-escalation-forbidden.md`、`reports/working/20260502-1140-forbidden-scope-fix.md`、現行 `/questions` API 実装。
- 背景: RAG が根拠不足で回答できない質問を、人手確認の業務フローへ引き継げるようにするため。
- 背景: 通常利用者の問い合わせ送信後に担当者向け一覧取得へ依存すると、不要な 403 により送信失敗のように見える問題があった。

## 要件の目的・意図

- 意図: RAG の回答不能を利用者の行き止まりにせず、担当者対応へエスカレーションできる状態にする。
- 意図: 通常利用者の問い合わせ作成導線を、担当者向け管理権限から分離する。
