## 3. 検討・判断したこと

- `ChatRunEventsStreamFunction` は Hono middleware を通らないため、handler 内で 200 と plain error response の両方に CORS header を持たせた。
- Lambda timeout や init error は worker 内 catch では拾えないため、Step Functions の Catch で failure marker Lambda を呼ぶ構成にした。
- failure marker は service method `markChatRunFailed()` に集約し、直接 worker が失敗した場合と Step Functions が失敗を検知した場合で同じ永続化動作になるようにした。
- `event.data` がある場合でも UI が進捗文言を読めるよう、SSE payload では `stage` / `message` と `data` object を併合する方針にした。
- 実 AWS deploy と実ブラウザ疎通はこの環境では未実施のため、単体/infra test と docs に未実施リスクを明示する。

## 4. 実施した作業

- `chat-run-events-stream.ts` に CORS header、plain/stream response header、payload builder、handler factory を追加した。
- `chat-run-mark-failed.ts` を追加し、Step Functions Catch から run failure を永続化できるようにした。
- `ChatRun` schema/type と `MemoRagService.startChatRun()` / `executeChatRun()` に `strictGrounded`、`useMemory`、`maxIterations` を追加した。
- Infra stack に `ChatRunMarkFailedFunction` と `ChatRunMarkFailedTask` を追加し、`ChatRunWorkerTask` の `Catch` へ接続した。
- Web hook に `timeout` event 後の最大 3 回再接続と `Last-Event-ID` 引き継ぎを追加した。
- API/Web/Infra のテストと infra snapshot を更新した。
- `DES_API_001.md` に CORS、reconnect、failure marker、ChatInput option の設計を追記した。
