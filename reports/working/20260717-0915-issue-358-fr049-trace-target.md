## 受けた指示

## 要件整理と判断

- 新規 `/chat` / `/chat-runs` の同期チャット親 trace は `chat_orchestration_run` を canonical とする。
- standalone search および検索子 trace は `rag_run` を維持する。
- `targetType` 欠落 legacy v1 は bounded default `rag_run` とし、runId / step からの推測再分類や persisted object の一括 rewrite は行わない。
- `targetType` は分類 metadata であり、認証・認可の根拠に使用しない。debug API permission、tenant partition、redaction / sanitizer 境界は維持する。
- shared contract が `targetType` を持たず oRPC parse で落とす不整合も同時に正本化する。

## なぜなぜ分析の要約

generic debug trace enum に `rag_run` と `chat_orchestration_run` の両方がある一方、chat orchestration producer 専用の canonical target contract と回帰テストがなかった。そのため実装方式由来の `rag_run` が残り、FR-049 の domain 境界と producer / fixture / shared contract が乖離した。

## 実施作業と成果物

- API type/schema と package contract に共通の target vocabulary、canonical chat target、legacy default を定義した。
- chat orchestration producer を `chat_orchestration_run` に修正し、search producer は `rag_run` のままであることを確認した。
- persistence/read の legacy fallback を名前付き bounded default に統一した。
- API/package vocabulary parity、canonical、unknown reject、legacy default、persisted legacy read、normal chat、async chat、permission-revoked の回帰テストを追加・強化した。
- FR-090 の unauthorized テストが実際の旧 producer で空集合を確認していたため、search run 以外の trace 残存を直接検査するよう修正した。
- `REQ_FUNCTIONAL_049.md`、`ARC_ADR_007.md`、`DES_API_002.md`、requirements coverage、source-backed generated API docs を同期した。
- ADR 番号は open PR #371 で `ARC_ADR_006.md` が予約済みのため 007 を使用した。
