# 作業完了レポート: 機能要求分類の再構成

保存先: `reports/working/20260504-0037-functional-requirements-restructure.md`

## 1. 背景

PR #93 では、MemoRAG MVP の機能要求を L0-L3 の分類体系で読み取れるようにする必要があった。初期案では上位索引に分類を追加したが、レビュー後、個別 `FR-*` ファイルからも分類を追えるようにする方針へ調整した。

## 2. 修正方針

- `REQUIREMENTS.md` は上位索引に限定する。
- `01_機能要求_FUNCTIONAL/README.md` を L0-L3 分類索引として追加する。
- 個別 `REQ_FUNCTIONAL_*` には分類メタデータを追記する。
- L1 主分類と L2 主機能群は各 `FR-*` で 1 つだけにする。
- 横断的な関係は関連カテゴリとして分離する。

## 3. 修正の要点

| 観点 | 修正内容 |
| --- | --- |
| 主分類 | `FR-*` ごとに L1 と L2 を 1 つだけ設定した。 |
| 関連カテゴリ | `FR-005`, `FR-020`, `FR-023`, `FR-024`, `FR-027` を関連カテゴリ付き要件として整理した。 |
| L2 の粒度 | `FR-021`, `FR-022`, `FR-025`, `FR-027` のような複合操作は 1 つの主機能群へ集約した。 |
| トレーサビリティ | `REQ_CHANGE_001.md` で L1 主カテゴリ、主分類の要求、関連カテゴリを確認できるようにした。 |

## 4. 分類上の採用ルール

```text
L0: rag-assist / MemoRAG MVP 機能要件
L1: 8 大カテゴリ
L2: 主機能群
L3: 個別 FR
```

各 `FR-*` は L1 / L2 主分類を 1 つだけ持つ。1 つの要件に複数の操作が含まれる場合、それらは要件本文や受け入れ条件で扱い、分類索引上の L2 行には分割しない。

## 5. 検証

- `REQ_FUNCTIONAL_001.md` から `REQ_FUNCTIONAL_028.md` までに `分類（L0-L3）` セクションがあることを確認した。
- 分類索引の主分類マップで `FR-001` から `FR-028` までが各 1 回だけ出現することを確認した。
- 関連カテゴリ表は主分類の重複ではなく、影響確認用の補助情報として記載した。

## 6. 残リスク

- 将来 `FR-*` を追加する場合、分類索引、個別要件ファイル、変更管理トレーサビリティを同時更新する必要がある。
- 機械的な整合性検証を強化する場合は、分類情報を JSON/YAML へ切り出す余地がある。
