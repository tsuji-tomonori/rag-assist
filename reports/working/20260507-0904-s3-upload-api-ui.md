## 1. 受けた指示

- 主な依頼: S3 を介したアップロードを API として用意し、UI 側もその API を使う形へ変更する。
- 背景: Allganize PDF コーパスの benchmark seed で API Gateway/Lambda の 10MiB body limit により `HTTP content length exceeded 10485760 bytes` が発生していた。
- 条件: 認証・認可境界を維持し、benchmark seed と通常ドキュメントアップロードを混在させない。

## 3. 検討・判断したこと

- 直接 `/documents` に base64 を投げる方式は小さいテキスト/Markdown 互換として残し、PDF などの大きいファイルは `/documents/uploads` のセッション作成、S3 への PUT、ingest 確定の 3 段階へ分けた。
- ローカル開発では S3 presigned URL が使えないため、同じ API contract で `/documents/uploads/{uploadId}/content` にアップロードできる fallback を用意した。
- 通常アップロードと benchmark seed は `purpose`、prefix、metadata validation、permission で分離し、`benchmark:seed_corpus` を持たない利用者が benchmark seed を作れないようにした。
- CodeBuild の post_build は BUILD 失敗時でも最低限の空成果物を生成して S3 upload できるようにし、本来の失敗原因を隠す二次失敗を減らした。

## 4. 実施した作業

- API に upload session 作成、upload content fallback、uploaded object ingest の route と schema を追加。
- S3/local object store adapter に bytes read/write と presigned upload URL 作成を追加。
- Web UI のファイルアップロード hook/API を upload session 経由へ変更。
- benchmark corpus seed の PDF アップロードを upload session 経由へ変更。
- CDK の documents bucket に CORS と `uploads/` lifecycle を追加し、CodeBuild post_build fallback artifact を追加。
- API/Web/benchmark/infra のテストと snapshot、README/API examples/operations docs を更新。
